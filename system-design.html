Backend Services
Microservice responsibilities, APIs, and internal communication
Responsibilities
User Registration
Validate input → hash password (bcrypt, 12 rounds) → generate email verification token → store in PostgreSQL → send verification email via SES
Login
Verify credentials → issue short-lived Access Token (JWT, 15min) + long-lived Refresh Token (httpOnly cookie, 30 days) stored in Redis
Token Refresh
Validate refresh token from Redis → rotate (issue new pair, revoke old) → sliding window expiry
OAuth2 (Google)
Receive code from client → exchange with Google → upsert user record → issue JWT pair same as password login
Password Reset
Generate signed token (HMAC-SHA256, 15min TTL) → store hash in Redis → email link → verify → update password → invalidate all sessions
JWT Payload Structure
{ "sub": "user_uuid", "iat": 1710000000, "exp": 1710000900, // +15min "role": "user", "jti": "unique-token-id" // for revocation } // Signed with RS256 (asymmetric) // Public key exposed at /.well-known/jwks.json
⚠️ Refresh tokens are stored in Redis with the user's device fingerprint. Logging out on one device doesn't expire other sessions.
Message Send Flow
Client sends via WebSocket
Event: message:send — encrypted payload, temp client ID, conversation ID
Server validates & persists
Auth middleware verifies JWT → write to MongoDB with status "sent" → generate server message ID
ACK to sender
Emit message:ack back with server ID → client swaps temp ID, marks ✓ sent
Publish to Redis
Publish to channel chat:{conversationId} — all WS nodes subscribed to that channel receive it
Push to recipient
If online: WebSocket delivery → receipt ✓✓. If offline: Kafka event → Notification Service → FCM/APNs push

Profile Management
CRUD for display name, avatar (uploaded to S3 with pre-signed URL), bio, and privacy settings. Avatar thumbnails generated via Lambda@Edge on upload.
Contact Discovery
Search by username or phone (hashed phone lookup preserving privacy). Contact requests with accept/reject. Block list stored in PostgreSQL with cascading privacy rules.
Presence System
On WS connect → SET presence:{userId} = "online" in Redis with 60s TTL. Heartbeat every 30s resets TTL. On disconnect → publish "offline" event to all contacts.
Upload Flow
Client requests upload URL
POST /media/upload-url → server validates file type/size → generates S3 pre-signed PUT URL (5min expiry)
Direct S3 upload
Client uploads directly to S3 — bypasses backend, saving bandwidth. Encrypted client-side before upload for E2EE media.
Post-upload processing
S3 event → Lambda: generate thumbnails (images/video), transcode video to HLS, extract metadata, run virus scan
CDN delivery
Media served via CloudFront with signed URLs (time-limited). Original S3 bucket is private; access only via CDN.
Supported Media

Images JPG, PNG, WebP, GIF — max 10MB
Video MP4, MOV — max 100MB — HLS streaming
Audio MP3, OGG, M4A — max 25MB
Docs PDF, DOCX, XLSX — max 50MB

💡 Images are compressed with Sharp on Lambda. WebP conversion saves ~30% bandwidth.
🔥 Firebase (FCM)
Android + Web push notifications. Batched via Kafka consumer. Handles delivery failures with exponential backoff. Device tokens stored per-user in PostgreSQL.
🍎 Apple (APNs)
iOS push notifications. Certificate-based auth. Background pushes for message sync. Silent notifications for read receipts. Notification grouping by conversation.
📧 Email (SES)
Transactional emails: verification, password reset, login alerts. Templated with AWS SES. Bounce/complaint handling via SNS. Daily digest for missed messages (opt-in).

Database Design
Schema, sharding strategy, and data access patterns
🐘 PostgreSQL — Relational Data
-- Users table CREATE TABLE users ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), username VARCHAR(30) UNIQUE NOT NULL, email VARCHAR(255) UNIQUE NOT NULL, password_hash TEXT, avatar_url TEXT, created_at TIMESTAMPTZ DEFAULT now(), last_seen TIMESTAMPTZ ); -- Conversations CREATE TABLE conversations ( id UUID PRIMARY KEY, type VARCHAR(10), -- 'direct' | 'group' name TEXT, -- null for direct created_by UUID REFERENCES users(id) ); -- Members CREATE TABLE conversation_members ( conversation_id UUID REFERENCES conversations(id), user_id UUID REFERENCES users(id), role VARCHAR(10) DEFAULT 'member', joined_at TIMESTAMPTZ DEFAULT now(), PRIMARY KEY (conversation_id, user_id) );
🍃 MongoDB — Message Storage
// messages collection { _id: ObjectId, conversationId: "uuid", senderId: "uuid", type: "text|image|video|file", // E2EE: ciphertext stored, not plaintext content: { ciphertext: "base64...", iv: "base64...", algorithm: "AES-256-GCM" }, mediaUrl: "https://cdn....", // if media status: "sent|delivered|read", replyTo: ObjectId, // quoted msg reactions: [{ emoji, userId }], editedAt: ISODate, deletedAt: ISODate, // soft delete createdAt: ISODate, // Indexed fields // idx: { conversationId: 1, createdAt: -1 } // idx: { conversationId: 1, status: 1 } }
🔴 Redis Data Structures
# Session store SET session:{jti} {userId} EX 2592000 # Presence SET presence:{userId} "online" EX 60 ZADD online_users {timestamp} {userId} # Unread counts HINCRBY unread:{userId} {convId} 1 # Rate limiting INCR ratelimit:{userId}:{minute} EXPIRE ratelimit:{userId}:{minute} 60 # Message dedup SET dedup:{clientMsgId} {serverId} EX 300 # Typing indicator SET typing:{convId}:{userId} 1 EX 5
📊 Sharding Strategy
MongoDB — shard by conversationId. All messages in a conversation land on the same shard → efficient range queries.

PostgreSQL — partition users table by id % N (hash partitioning). Read replicas per region for low-latency reads.

Redis Cluster — 6 nodes (3 primary, 3 replica). Hash slots by key prefix. Cluster-aware clients.
🔁 Data Retention
Active messages — MongoDB primary (hot tier)
6 months+ — tiered to MongoDB Atlas archive
2 years+ — compressed to S3 Glacier

Deletion — soft delete (deletedAt field) for 30 days, then hard delete by cron.

GDPR erasure — wipes user PII across all collections within 30 days of request.

Security Architecture
Encryption, authentication, and attack prevention
🔐 End-to-End Encryption
🗝️
X3DH Key Agreement
Extended Triple Diffie-Hellman for initial session establishment. Identity keys, signed pre-keys, and one-time pre-keys published to the server. Server never sees private keys.
🔄
Double Ratchet Algorithm
Forward secrecy + break-in recovery. New encryption key per message. Compromising one key doesn't expose past or future messages.
🛡️
Transport Encryption
TLS 1.3 for all HTTP/WS connections. Certificate pinning in mobile apps. HSTS with 1-year max-age + preload.
🛡️ Attack Prevention
🚦
Rate Limiting
Token bucket per user, per IP. Login: 5 attempts/min, lockout 15min. Message send: 60/min. File upload: 10/min. Implemented in Redis at API gateway.
🧹
Input Sanitisation
All user input validated with Zod schemas. HTML stripped server-side. SQL injection impossible (parameterised queries, ORM). XSS prevented via DOMPurify on client.
🦠
Media Scanning
All uploads scanned with ClamAV on Lambda before CDN distribution. MIME type verification (magic bytes, not extension). Max file size enforced at gateway.
🔑
Secrets Management
All secrets in AWS Secrets Manager. Zero hardcoded credentials. Automatic rotation for DB passwords every 30 days. Env vars injected at runtime via ECS task definitions.

🧩 Bottleneck Analysis
WebSocket connections — ~50k persistent connections per node (Node.js event loop). Scale via Kubernetes HPA. Redis Pub/Sub decouples nodes.

Message fan-out (group chats) — For groups >1000 members, async fan-out via Kafka. Online members get WS delivery; others get push notifications in batches.

Media bandwidth — Handled entirely by S3 + CloudFront. Backend never proxies media bytes.

DB write throughput — MongoDB Atlas M40+, writes to primary, reads from secondaries. Bulk inserts for batch operations.
💡 Cost optimisation: Spot instances for stateless pods save ~60%. Reserved instances for databases. S3 Intelligent-Tiering automatically moves old media to cheaper storage. Estimated cost at 100k DAU: ~$800/month AWS.
